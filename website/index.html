What this is

Old pages carry DOM XSS in code nobody will rewrite. DOMFortify turns on Trusted Types with a CSP and registers a single default policy for the whole realm: HTML sinks (innerHTML, insertAdjacentHTML, document.write) are routed through a sanitizer; script sinks (eval, javascript: URLs, script.src) are refused outright - code has no safe subset. No application code changes. Trusted Types is Baseline across current browsers as of 2026.

Honest edges, all visible below: enforcement comes from the CSP, not the library; with no sanitizer it fails closed (sinks throw, never leak); the first script to claim default wins, so load fortify.js first; it guards only the realm it loads in (see the child frame); and it covers only Trusted Types sinks - inline handlers, style, and direct URL properties stay open and need a script-src without 'unsafe-inline'.

Child frame - a separate realm, DOMFortify installed into it too

probed on first write

DOMFortify guards only the realm it loads in, so a bare child frame is unprotected. Under an enforcing parent (common on a hardened site) an unprotected frame makes document.write throw - it wants a trusted value and finds no policy. The fix is the per-realm rule itself: install DOMFortify in the frame too, which the demo does here - so the write is sanitized by the frame's own policy.
